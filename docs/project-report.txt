WASHFLOW PROJECT REPORT
=======================

Project Title
-------------
WashFlow: Laundry Management Website

Problem Statement
-----------------
Laundry service coordination often becomes messy when customers, washing center
staff, and administrators rely on disconnected channels for requests, updates,
status tracking, and issue handling. WashFlow solves this by centralizing the
request lifecycle in one role-based web application.

Solution Approach
-----------------
WashFlow was built as a full-stack website using React for the frontend and a
TypeScript + Node.js + Express + MongoDB backend. The solution follows a layered
architecture so that presentation, HTTP orchestration, business rules, and data
access remain separated. This improves maintainability, scalability, and testing.

System Design Optimization
--------------------------
1. Layered architecture:
   Request orchestration is separated from business logic and database access.
   This reduces coupling and makes features easier to extend.

2. Repository isolation:
   Database queries are isolated inside repository classes. This keeps the
   services focused on business rules and makes repository replacement easier.

3. Role-based query filtering:
   Customer, manager, and admin data views are filtered at the service/repository
   level so the frontend receives only the relevant records.

4. Notification factory:
   Notification creation is centralized to reduce duplicated branching logic and
   standardize messaging across workflows.

5. Composition root:
   Dependencies are wired once in backend/src/container/index.ts so services and
   controllers receive their collaborators in a consistent way.

OOP Concepts Used
-----------------
1. Encapsulation
   - Business rules are encapsulated in services:
     backend/src/services/auth.service.ts
     backend/src/services/laundry-request.service.ts
     backend/src/services/concern.service.ts
     backend/src/services/notification.service.ts

2. Abstraction
   - Repository interfaces define contracts:
     backend/src/repositories/interfaces/*

3. Inheritance
   - The conceptual domain layer under root src/ uses shared entity abstractions:
     src/core/Entity.ts
     src/core/Auditable.ts

4. Polymorphism
   - Multiple repository implementations conform to the same repository contract
     style and can be injected into services through shared interfaces.

Design Patterns Implemented
---------------------------
1. Repository Pattern
   Why:
   - To separate business logic from direct Mongoose queries
   - To make data access more maintainable and replaceable

   Where:
   - backend/src/repositories/user.repository.ts
   - backend/src/repositories/washing-center.repository.ts
   - backend/src/repositories/laundry-request.repository.ts
   - backend/src/repositories/concern-ticket.repository.ts
   - backend/src/repositories/notification.repository.ts

2. Factory Pattern
   Why:
   - To centralize notification payload creation
   - To avoid scattered conditional notification logic

   Where:
   - backend/src/factories/notification.factory.ts

3. Dependency Injection / Composition Root
   Why:
   - To wire dependencies in one place
   - To keep route/controller/service creation explicit

   Where:
   - backend/src/container/index.ts

SOLID Principles
----------------
1. Single Responsibility Principle
   - Routes map endpoints only
   - Controllers manage HTTP flow only
   - Services contain business rules only
   - Repositories contain persistence logic only

   Examples:
   - backend/src/routes/requests.ts
   - backend/src/controllers/laundry-request.controller.ts
   - backend/src/services/laundry-request.service.ts
   - backend/src/repositories/laundry-request.repository.ts

2. Open/Closed Principle
   - Notification creation can be extended by adding new factory methods without
     rewriting controller logic
   - New roles or workflows can be layered through services and controllers

   Example:
   - backend/src/factories/notification.factory.ts

3. Liskov Substitution Principle
   - Services rely on repository contracts rather than Mongoose details
   - Repository implementations can be substituted as long as they respect the
     interface behavior

   Example:
   - backend/src/repositories/interfaces/*

4. Interface Segregation Principle
   - Repositories are split by domain responsibility instead of using one giant
     data-access interface

   Example:
   - user, request, concern, notification, and washing center repository
     interfaces are separate

5. Dependency Inversion Principle
   - Services depend on abstractions (repository interfaces)
   - Concrete implementations are wired in the container

   Example:
   - backend/src/services/*.ts
   - backend/src/container/index.ts
